# Architecture Design

## Executive Summary

This document describes the architecture of a private Confluent Cloud Kafka platform deployed on Microsoft Azure using Terraform.

The solution is designed for an Energy Platform responsible for ingesting smart meter telemetry and operational grid events. Kafka serves as the central event streaming backbone, enabling decoupled communication between producers and consumers while maintaining secure and private network connectivity.

The architecture leverages Azure PrivateLink to establish private connectivity between AKS workloads and Confluent Cloud Kafka brokers. Kafka traffic remains on private network paths and never traverses the public Internet.

The design emphasizes:

* Security and network isolation
* Infrastructure as Code (Terraform)
* Least privilege access control
* Operational observability
* Scalability and maintainability

---

# Business Use Case

The platform receives telemetry and operational events from field devices such as smart meters, substations, and grid monitoring systems.

Kafka is used as the event backbone to transport these events to downstream analytics and operational systems.

## Topic: meter-readings

Stores smart meter telemetry.

Example payload:

```json
{
  "meterId": "MTR-1001",
  "timestamp": "2026-06-01T10:15:00Z",
  "consumptionKwh": 5.8
}
```

## Topic: grid-alerts

Stores operational alerts generated by substations and monitoring systems.

Example payload:

```json
{
  "siteId": "SUBSTATION-10",
  "severity": "HIGH",
  "event": "Voltage Spike"
}
```

# Azure Components

## Resource Group

Resource Group:

```text
rg-confluent-mgmt-poc
```

Provides a logical management boundary for all Azure resources associated with this solution.

---

## Virtual Network

Virtual Network:

```text
vnet-confluent-mgmt-poc
```

Address Space:

```text
10.0.0.0/16
```

Provides network isolation and connectivity between AKS workloads and Azure Private Endpoints.

---

## Subnets

### AKS Subnet

```text
snet-aks
10.0.2.0/24
```

Hosts AKS worker nodes and application workloads.

### Private Endpoint Subnet

```text
snet-private-endpoint
10.0.3.0/24
```

Hosts Azure Private Endpoints used to access Confluent Cloud through PrivateLink.

---

## AKS Cluster

AKS hosts producer and consumer workloads responsible for publishing and consuming Kafka events.

Example services:

* meter-ingestion-service
* grid-analytics-service

---

## NAT Gateway

The NAT Gateway provides controlled outbound internet access for AKS workloads.

Typical use cases include:

* Container image pulls
* Software package downloads
* External API access

Kafka traffic does not use the NAT Gateway.

---

## Private DNS Zone

The Private DNS Zone provides private name resolution for Confluent Cloud PrivateLink endpoints.

Expected resolution:

```text
pkc-xxxxx.azure.confluent.cloud
```

returns:

```text
10.x.x.x
```

instead of a public IP address.

---

## Azure Private Endpoint

The Private Endpoint provides a private IP address within the VNet and acts as the network entry point to the Confluent Cloud Kafka brokers.

---

A Confluent Cloud Dedicated Cluster is used to support private networking through Azure PrivateLink.

## Topics

### meter-readings

Stores telemetry data generated by smart meters.

### grid-alerts

Stores operational events and alerts generated by grid infrastructure.

---

## Service Account

A dedicated service account is used by AKS workloads.

Example:

```text
sa-energy-platform
```

---

## API Key

Provides authentication for Kafka clients.

Used by:

* Producers
* Consumers

---

## ACLs

The following permissions are granted:

```text
READ  meter-readings
WRITE meter-readings

READ  grid-alerts
WRITE grid-alerts
```

Consumer group permissions:

```text
READ Consumer Groups
```

The design follows the principle of least privilege.

---

# Network Design

Kafka communication follows the path:

```text
AKS
  |
Private DNS
  |
Azure Private Endpoint
  |
Azure PrivateLink
  |
Confluent Kafka
```

No Kafka traffic traverses the public Internet.

Benefits include:

* Reduced attack surface
* Compliance alignment
* Private connectivity
* Predictable network paths

---

# Security Design

## Authentication

Kafka clients authenticate using:

* API Key
* API Secret

Protocol:

```text
SASL_SSL
```

---

## Authorization

Kafka ACLs restrict access to only the required topics and consumer groups.

No administrative permissions are granted to application workloads.

---

## Secrets Management

For this Proof of Concept:

* Kubernetes Secrets

Production recommendation:

* Azure Key Vault
* AKS Workload Identity

---

## Network Security

Security controls include:

* Azure PrivateLink
* Private Endpoint
* Private DNS
* VNet isolation

Public Kafka access is disabled.

# Design Decisions

## Why AKS?

AKS provides a managed Kubernetes platform capable of running scalable producer and consumer workloads.

---

## Why Confluent Cloud?

Confluent Cloud reduces operational overhead associated with running Kafka infrastructure while providing enterprise-grade Kafka capabilities.

---

## Why PrivateLink?

PrivateLink ensures Kafka traffic remains on private network paths and never traverses the public Internet.

---

## Why Terraform?

Terraform provides consistent, repeatable, and auditable infrastructure provisioning.

---

## Why Datadog?

Datadog provides centralized observability across infrastructure, Kubernetes workloads, and Kafka-connected applications.

---

# Conclusion

This architecture provides a secure, private, and scalable event streaming platform for energy telemetry and operational event processing. Azure PrivateLink and Private DNS ensure all Kafka traffic remains private, while Terraform provides Infrastructure-as-Code automation and repeatability. The design emphasizes security, operational simplicity, and observability while remaining extensible for future production requirements.
