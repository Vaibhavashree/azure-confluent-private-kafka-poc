# Architecture Overview

## Executive Summary

This Proof of Concept (PoC) demonstrates the design and Terraform-based provisioning approach for a private Apache Kafka platform using Confluent Cloud and Azure Kubernetes Service (AKS).

The solution enables AKS-hosted applications to securely publish and consume Kafka messages through private networking without exposing Kafka traffic to the public Internet.

The architecture is designed for an energy management use case where applications process meter telemetry and grid operational events through Kafka topics.

---

# Business Use Case

An energy company requires a secure and scalable event streaming platform for:

* Smart meter telemetry ingestion
* Grid monitoring and alert processing
* Near real-time analytics workloads
* Event-driven microservices running on AKS

To satisfy security requirements, Kafka traffic must remain private and avoid public network exposure.

# Core Components

## Azure Components

### Resource Group

Logical container for all Azure resources.

**Resource:**

```text
rg-confluent-mgmt-poc
```

---

### Virtual Network

Provides network isolation for AKS and PrivateLink connectivity.

**Resource:**

```text
vnet-confluent-mgmt-poc
```

---

### AKS Cluster

Hosts producer and consumer workloads responsible for processing energy-related events.

**Configuration**

* Azure CNI networking
* System Assigned Managed Identity
* Standard_B2s worker nodes
* Dedicated AKS subnet

---

### Azure Private Endpoint

Provides private connectivity between Azure and Confluent Cloud.

Traffic remains on Microsoft's private backbone network and does not traverse the public Internet.

---

### Private DNS Zone

```text
privatelink.confluent.cloud
```

Used to resolve Confluent broker endpoints to private IP addresses.

---

### NAT Gateway

Provides controlled outbound Internet access for AKS workloads.

Kafka traffic does not use the NAT Gateway.

Kafka communication is routed through Azure PrivateLink.

---

# Confluent Cloud Components

## Environment

Logical isolation boundary for Kafka resources.

---

## Dedicated Kafka Cluster

A dedicated Confluent Cloud Kafka cluster deployed in Azure East US 2.

Characteristics:

* Dedicated deployment model
* Private networking architecture
* Single-zone deployment for PoC
* Terraform managed

---

## Kafka Topics

### meter-readings

Stores telemetry events received from smart meters.

Example payload:

```json
{
  "meterId": "MTR-1001",
  "timestamp": "2025-07-01T12:00:00Z",
  "usage": 125.5
}
```

---

### grid-alerts

Stores operational alerts generated by grid monitoring systems.

Example payload:

```json
{
  "alertId": "ALT-001",
  "severity": "HIGH",
  "message": "Transformer temperature threshold exceeded"
}
```

---

## Service Account

Dedicated service account used by AKS workloads.

```text
sa-energy-platform
```

---

## API Key

Kafka API Key and Secret associated with the service account.

Applications authenticate using SASL_SSL.

---

## ACLs

Least-privilege ACLs restrict access to:

### meter-readings

* READ
* WRITE

### grid-alerts

* READ
* WRITE

### Consumer Groups

* READ

# Private Networking Design

The architecture follows a private connectivity model.

## Azure Side

Implemented through Terraform:

* Virtual Network
* Private Endpoint Subnet
* Azure Private Endpoint
* Private DNS Zone
* Private DNS Link

## Confluent Side

Represented in architecture and Terraform comments:

* Confluent Network
* PrivateLink Access
* PrivateLink Attachment

Together these components provide end-to-end private connectivity between AKS and Confluent Cloud.

# Security Design

## Network Security

* PrivateLink connectivity
* No public Kafka endpoints
* Private DNS resolution
* Network isolation via VNet

## Authentication

Kafka clients authenticate using:

* API Key
* API Secret
* SASL_SSL

## Authorization

Access controlled through Kafka ACLs.

Principle of least privilege is enforced.

# Conclusion
