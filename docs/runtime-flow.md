# Runtime Traffic Flow

## Overview

This document describes the end-to-end flow of data between Azure Kubernetes Service (AKS) workloads and Confluent Cloud Kafka.

The platform processes telemetry and operational events generated by energy infrastructure and transports them securely through Kafka using Azure PrivateLink.

---

# Producer Flow

## Step 1 – Event Generation

A smart meter generates a telemetry event.

Example:

```json
{
  "meterId": "MTR-1001",
  "timestamp": "2026-06-01T10:15:00Z",
  "consumptionKwh": 5.8
}
```

---

## Step 2 – Producer Service

The event is sent to the producer application running inside AKS.

Example Service:

```text
meter-ingestion-service
```

---

## Step 3 – DNS Resolution

The producer resolves the Kafka bootstrap endpoint.

```text
pkc-xxxxx.azure.confluent.cloud
```

The Private DNS Zone returns a private IP address associated with the Azure Private Endpoint.

---

## Step 4 – Private Connectivity

Traffic flows through:

```text
AKS
  |
Private Endpoint
  |
Azure PrivateLink
  |
Confluent Kafka
```

No traffic traverses the public Internet.

---

## Step 5 – Authentication

The Kafka broker validates:

* API Key
* API Secret

Protocol:

```text
SASL_SSL
```

---

## Step 6 – Authorization

Kafka ACLs verify topic permissions.

Example:

```text
WRITE meter-readings
```

---

## Step 7 – Message Persistence

The event is written to:

```text
meter-readings
```

---

# Consumer Flow

## Step 1 – Consumer Service

Consumer application:

```text
grid-analytics-service
```

subscribes to:

```text
meter-readings
```

---

## Step 2 – Message Consumption

The consumer reads telemetry events from Kafka.

---

## Step 3 – Analytics Processing

The data is processed and forwarded to analytics and reporting systems.

---

# Security Controls

The runtime flow incorporates:

* PrivateLink
* Private DNS
* SASL_SSL
* Service Accounts
* API Keys
* Kafka ACLs

All communication remains encrypted in transit.
